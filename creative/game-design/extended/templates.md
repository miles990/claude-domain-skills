```markdown
# Game Design Document

## 1. 概述

- 主要玩法

## 3. 進度系統
- 成長曲線
- 解鎖機制

## 4. 數值系統
- 資源類型

- 經濟模型

## 5. 內容規劃
- 關卡設計
- 敵人/NPC
- 物品/裝備
```

## roguelike-design

```markdown
## Roguelike 核心設計

### 隨機性平衡
| 元素 | 隨機程度 | 說明 |
|------|----------|------|
| 地圖佈局 | 高 | 程序生成 |
| 敵人配置 | 中 | 難度遞增 |
| 物品掉落 | 中-高 | Build 多樣性 |
| Boss 機制 | 低 | 可學習模式 |
| 核心機制 | 無 | 玩家可掌握 |

### Meta 進度設計
| 單局內 | 跨局（永久）|
|--------|-------------|
| 獲得強化道具 | 解鎖新角色/武器 |
| 臨時能力提升 | 永久屬性加成 |
| 資源累積 | 起始資源增加 |
| 局內成長感 | 長期目標 + 多樣性 |

### Build 設計
- 每個道具獨立有用
- 道具組合產生協同效應 (Synergy)
- 極端 Build 應該可行但有風險
- 避免「必選」道具（所有選項都應有價值）
```

## action-game-design

### 動作遊戲核心要素

**操作手感**: 60fps、輸入延遲 < 100ms

**打擊感公式**:
1. 攻擊命中 -> 頓幀 2-4 幀
2. 同時播放打擊音效 + 粒子特效
3. 敵人播放受傷動畫 + 擊退
4. 相機輕微震動（可選）

**敵人設計**: 清晰的攻擊前搖、可讀性

## puzzle-game-design

### 謎題設計原則

1. **漸進式複雜度**: 簡單 -> 變體 -> 組合 -> 反轉
2. **公平性**: 所有線索在謎題前已提供、解法符合邏輯、避免「pixel hunting」
3. **「啊哈」時刻**: 難度 = 資訊隱藏程度 x 邏輯步驟數

### 提示系統
- 等級 1：方向性提示「注意那個角落」
- 等級 2：具體提示「試著移動那個箱子」
- 等級 3：直接解答「把箱子推到開關上」

## mobile-game-design

### 手遊設計考量

**操作限制**: 單手可玩優先、虛擬搖桿避免太複雜、點擊優於滑動

**遊戲時長**: 單局 3-5 分鐘（碎片化）、可隨時暫停/儲存、離線獎勵機制

### 變現模式
| 模式 | 說明 |
|------|------|
| 付費買斷 | Premium，一次付費 |
| 內購 (IAP) | 道具、外觀、VIP |
| 廣告 | 激勵廣告、插頁廣告 |
| 訂閱 | 月費會員特權 |
| Battle Pass | 賽季通行證 |

**付費設計原則**: 付費 != 獲勝、免費玩家也要有良好體驗、付費內容要有明確價值感

## narrative-structure

### 敘事結構類型

1. **線性敘事**: A -> B -> C -> D -> 結局（故事緊湊，開發成本低，重玩價值低）
2. **分支敘事**: 多條路線到多結局（玩家有選擇感，內容開發量大）
3. **聚合式分支**: 分支後收斂到主線（有選擇感但收斂，玩家可能感覺選擇無意義）
4. **開放世界敘事**: 主線 + 多條支線並行，玩家自由選擇順序

### 對話系統設計

**好的選項**: 性格鮮明、有明確後果、反映玩家意圖

**避免**: 三個選項說同樣的話、假選擇、選項文字與實際對話不符

**對話節奏**: 單句對話 <= 2 行、重要資訊要有視覺強調、允許跳過但保留重要資訊

## multiplayer-design

### 網路模型比較

| 模型 | 說明 | 適用 | 延遲容忍 |
|------|------|------|----------|
| P2P | 玩家直連 | 格鬥、賽車 | 低 |
| Client-Server | 中央伺服器 | MMO、FPS | 中 |
| Rollback | 預測+回滾 | 格鬥遊戲 | 最低 |
| Lockstep | 同步輸入 | RTS | 中 |

### 延遲補償技術
- 客戶端預測（本地先執行）
- 伺服器權威（Server Authoritative）
- 插值顯示（Interpolation）
- 命中判定回溯（Lag Compensation）

### 匹配系統 (Elo/MMR)

**公式**: 新 Rating = 舊 Rating + K x (實際結果 - 預期結果)

**K 值**: 新玩家 K=32（快速調整）、老玩家 K=16（穩定）

**保護機制**: 新手保護、連敗保護、段位保護

## live-service-design

### 內容更新節奏

| 週期 | 內容 |
|------|------|
| 週更新 | 每日任務刷新、週常活動、商店輪換 |
| 月更新 | 新角色/武器、限時活動、Battle Pass 賽季 |
| 季更新 | 大型版本更新、新遊戲模式、主線劇情 |

### 營運指標 (KPI)

**用戶指標**:
- DAU/MAU 比: 健康值 20-30%
- D1/D7/D30 留存: 40%/20%/10%
- Stickiness: DAU/MAU > 20%

**營收指標**:
- ARPU = 總收入/總用戶
- ARPPU = 總收入/付費用戶
- LTV = ARPU x 生命週期

**健康公式**: LTV > CPI x 3

## development-workflow

### 遊戲開發生命週期

| 階段 | 時間 | 產出 |
|------|------|------|
| 概念期 | 1-2 週 | 遊戲概念文檔、市場研究、可行性評估 |
| 原型期 | 2-4 週 | 核心機制原型、玩法驗證、技術風險評估 |
| 前製期 | 1-3 個月 | GDD 完整化、美術風格、垂直切片 |
| 製作期 | 6-18 個月 | 內容製作、迭代優化、Alpha/Beta |
| 後製期 | 1-2 個月 | 除錯、優化、認證 |
| 上市後 | 持續 | 營運活動、內容更新、社群維護 |

### 敏捷開發 (Scrum)

**Sprint 週期**: 建議 2 週，每個 Sprint 要有可玩的產出

**DoD (Definition of Done)**: 功能完成、無已知 Bug、通過 QA、程式碼審查、文檔更新
