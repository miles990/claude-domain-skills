# Game Design Checklists

## level-design-checklist

### 教學設計
- [ ] 新機制有安全環境練習
- [ ] 複雜機制分步驟引入
- [ ] Show, don't tell

### 節奏控制
- [ ] 緊張與放鬆交替
- [ ] 探索與挑戰平衡
- [ ] 適當的檢查點

### 可讀性
- [ ] 路徑清晰可見
- [ ] 危險區域有視覺提示
- [ ] 目標明確

### 重玩性
- [ ] 可探索的秘密區域
- [ ] 多種通關路線
- [ ] 收集要素

## playtest-questionnaire

### 基本體驗
1. 遊戲好玩嗎？（1-10）
2. 你會想繼續玩嗎？
3. 一句話描述這遊戲

### 核心機制
4. 操作直覺嗎？
5. 有什麼讓你困惑的地方？
6. 最喜歡的部分是？
7. 最不喜歡的部分是？

### 難度感受
8. 難度合適嗎？
9. 有卡關的地方嗎？
10. 有太簡單無聊的地方嗎？

## playtest-observation

| 觀察 | 可能問題 |
|------|----------|
| 玩家停頓 | 不知道該做什麼 |
| 反覆嘗試 | 機制不直覺或太難 |
| 跳過教學 | 教學太囉嗦 |
| 快速離開 | 核心循環不吸引人 |

## economy-balance-checklist

### 經濟平衡檢查
- [ ] 設計有效的貨幣 Sink（強化失敗、修理費、消耗品）
- [ ] 限制每日獲取上限
- [ ] 使用綁定貨幣減少流通
- [ ] 定期審視流入/流出比例（目標 1:1）

## combat-balance-checklist

### 戰鬥數值平衡
- [ ] TTK (Time To Kill) 合理嗎？
- [ ] 高低等級差距是否過大？
- [ ] 是否有數值溢出風險？
- [ ] 各職業/角色是否平衡？

## anti-cheat-checklist

### 防作弊設計
- [ ] 關鍵邏輯在伺服器端
- [ ] 輸入驗證
- [ ] 資料加密
- [ ] 異常行為偵測

## tutorial-pyramid

### 教學設計金字塔（從上到下）

1. **強制教學** - 只用於最核心機制
2. **提示引導** - 視覺提示、箭頭
3. **環境引導** - 關卡設計暗示
4. **自由探索** - 玩家自行發現

### 原則
- 能用關卡設計教的，不用 UI 教
- 能用 UI 教的，不用文字教
- 讓玩家「做」而不是「看」
- 失敗也是學習（但要給明確反饋）

## scrum-dod

### Definition of Done (DoD)
- [ ] 功能完成
- [ ] 無已知 Bug
- [ ] 通過 QA 測試
- [ ] 程式碼審查
- [ ] 文檔更新

### 遊戲開發特殊考量
| 挑戰 | 解決方案 |
|------|----------|
| 難以估計工時 | 使用故事點而非小時 |
| 創意需要探索 | 留出「Spike」時間 |
| 跨部門依賴 | 明確的交付物規格 |
| 玩法驗證 | 每 Sprint 內部測試 |
